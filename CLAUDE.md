This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A **Karpathy-style "Context Stuffing" knowledge base** for Chinese-language **port smart-port (港口智慧化)** documents. Zero external vector DB by design: documents are *compiled* into high-density YAML artifacts at ingest time, and retrieval progressively stuffs context (ontology index → summaries → full text) into the LLM rather than doing ANN search. The whole system is driven by `ANTIGRAVITY.md` — treat that file as the schema/behaviour constitution; every YAML shape, doc-id format, and pipeline step is defined there.

There are three layers, all wrapping the same Python core:
- **`scripts/`** — the four CLI "engines" (the untouchable core, see guardrails below).
- **`app/` and `api/`** — two parallel FastAPI HTTP wrappers (see the important note below).
- **`frontend/`** — a Next.js (App Router) dashboard.

## Guardrails (from PROMPT_FOR_NEXT_SESSION.md — user-mandated)

1. **Do not rewrite the four core engines** in `scripts/` (`ingest`, `compile`, `search`, `relate`). All productization work wraps/encapsulates them — it does not replace their internals.
2. **Do not migrate to an external distributed vector DB.** The "zero-vector" Context Stuffing philosophy is the point. The optional embedding path in `search.py` (hybrid retrieval) only *boosts* BM25 ranking — it is not a replacement architecture.

## The core pipeline (read across `scripts/`)

The data lifecycle, with the directory each stage writes:

```
originals/ ──ingest──► raw/{doc_id}.txt + raw/{doc_id}.meta.yaml  (status: raw)
                              │
                              └──compile──► wiki/{doc_id}.summary.yaml     (Step A: LLM summary)
                                            meta/ontology/{doc_id}.ontology.yaml (Step B: ontology nodes)
                                            wiki/index.yaml                   (Step C: append index entry)
                                            meta/relations/{doc_id}.relations.yaml (Step D → relate.py)
                                            meta/relations/knowledge_graph.yaml
                                            .meta.yaml status → compiled
search reads wiki/index.yaml + summaries + raw/ full text (3 progressive layers)
```

- **doc_id format**: `doc_{YYYYMMDD}_{seq:03d}` (e.g. `doc_20260405_001`), generated by counting same-day meta files. Never hand-roll this; use `generate_doc_id()`.
- **Dedup**: `ingest.py` SHA256-hashes the source file and skips if the hash already exists in `wiki/index.yaml`.
- **`compile.py` Step D shells out to `relate.py` as a subprocess** (to avoid circular imports) — so compiling a doc also runs relation detection.
- **Scripts use module-level path constants** (`BASE_DIR`, `RAW_DIR`, `WIKI_DIR`, `INDEX_FILE`, etc.). Tests redirect these via `monkeypatch` (see `tests/conftest.py`) so the suite never touches real data. **Any new path a script reads must be a module constant** or test isolation breaks. The modular `app/` layer achieves the same isolation by reassigning `scripts.*` constants at call time (`app/utils/background.py`, `app/routers/qa.py`) or via `KB_BASE_DIR`.

### Search's three layers (`scripts/search.py`)

1. **Layer 1 — hybrid candidate filter** (`layer1_filter`): `BM25Engine` (jieba `cut_for_search` tokenization) **fused with** an optional `VectorEngine` (`scripts/embedding_client.py`) via **Reciprocal Rank Fusion** (k=60). BM25 hits are the *gate* — vector scores only re-rank existing BM25 candidates (or, if BM25 finds nothing, a high-vector-threshold fallback is used). Candidate set → Top-20.
2. **Layer 2 — LLM relevance scoring** (`layer2_score`): LLM scores each candidate's `abstract_short` 0–1, keeps ≥0.5, Top-5.
3. **Layer 3 — answer + citations** (`layer3_answer`): stuffs Top-5 docs' full text (40k-char budget) into the LLM, emits a Markdown answer with `[文档标题 · 章节]` citations and a `📎 来源` block.

The `/qa` SSE endpoint additionally streams a **thought-trace** before the answer. Its event sequence (in order) is: `thought`(step 1/2/3) → `source`(citations) → `entity`(hit doc IDs) → `delta`(×N LLM tokens) → `done`.

## ⚠️ Two parallel FastAPI apps — know which one you're editing

This repo has **two independent FastAPI entrypoints** that both wrap the same `scripts/` engines. A migration from monolith → modular is in progress; **they are not kept in sync**:

| | `api/main.py` (`api.main:app`) | `app/` (`app.main:app`) |
|---|---|---|
| Style | single-file monolith | routers + `app.config.Settings` + `app/schemas.py` |
| Routes | full paths inline: `/api/v1/wiki/index`, `/graph`, `/docs`, `/upload`, `/search`, `/search/stream`, `/qa`, `/lint` | `/api/v1` prefix via routers; typed `response_model`s; extra routes (`/ontology`, `/relate/{id}`, `/graph/{id}/relations`) |
| Tested by | `tests/test_api.py` | `tests/test_api_*.py`, `test_tokenizer.py` |
| **Talked to by frontend?** | **Yes** — `frontend/src/lib/api.ts` matches these endpoints | No |

When the task touches "the backend the frontend uses", edit `api/main.py`. When it touches the typed/modular layer or its deeper tests, edit `app/`. When asked to "add an endpoint", confirm which app — they have diverged (e.g. `app/` has `/ontology` and `/relate/{doc_id}` that `api/main.py` lacks).

## Configuration

`.env` at project root (see `.env.example`). Loaded by a hand-rolled parser in every module (no `python-dotenv`) via `os.environ.setdefault` — set values are **not** overridden, so real env vars win over `.env`.

- LLM: `OPENAI_API_KEY`, `OPENAI_BASE_URL`. Production uses **Aliyun DashScope** OpenAI-compatible mode (`…/compatible-mode/v1`) with **qwen-plus** models.
- Per-stage models: `COMPILE_MODEL` / `ONTOLOGY_MODEL` / `RELATE_MODEL` / `SEARCH_MODEL`.
- Optional embedding path: `EMBEDDING_API_KEY`, `EMBEDDING_BASE_URL`, `EMBEDDING_MODEL_NAME`.
- `RELATE_MAX_COMPARE` (default 50) caps how many existing docs relation-detection compares against.
- `KB_BASE_DIR` redirects the entire data root — used by `app.config.Settings` for test isolation.

**Model default drift to be aware of:** `app/config.py` and `app/utils/background.py` default to `qwen-plus`; the raw `scripts/*.py` engines and `.env.example` default to `gpt-4o`. `.env` overrides all of these, so production behaviour is consistent — but when running a script *without* `.env`, expect OpenAI defaults.

## Testing notes

- The pytest suite is **fully offline** — LLM clients are mocked (`mock_llm_client` fixture) and jieba may be mocked to `None` (the tokenizer falls back to whitespace split). No API key required for `pytest`.
- `tests/conftest.py` builds an isolated knowledge-base skeleton under `tmp_path` and exposes `patch_ingest_paths` / `patch_compile_paths` / `patch_search_paths` / `patch_relate_paths` fixtures that monkeypatch each engine's path constants. **Use these fixtures for any test that touches the engines** — do not read/write the real `raw/`, `wiki/`, `meta/` from tests.
- `scripts/lint.py` (`Linter`) is a knowledge-base health checker (orphan pages, missing concepts, LLM-detected contradictions) exposed at `POST /api/v1/lint`.
- `scripts/logger.py` appends an audit trail to `wiki/log.md` on every ingest/search/relate/lint action.
