## 一、实测驱动的方法(非假设)

Loop #5 前我假设瓶颈是"LLM 调用次数多"。**实测推翻了这个假设**:

- Layer2 已是批量单调用(9 候选 1 次调用),不是多次
- 真实瓶颈:**单次 LLM 调用慢**,因为 qwen3.6-plus 是**思考模型**——生成前有 ~30s"思考"阶段
- 发现真正杠杆:**关闭思考**(enable_thinking=False)

这是 systematic-debugging 的价值——测了才发现真因,没在错误假设上浪费工作量。

## 二、四项改进 + LIVE 实证

| 改进 | 机制 | LIVE 实证 |
|---|---|---|
| **Layer2/3 关思考** | enable_thinking=False(qwen3 思考模型) | Layer2 56s→14s(4x)、Layer3 首 token 35s→0.4s(70x) |
| **Layer3 真流式** | stream API 透传,/qa 不再"先全生成再切片" | 首 token 立即可见 |
| **Layer2 结果缓存** | (query,候选id) key,60s TTL,进程级 LRU | 命中 0.000s(实测除零证明) |
| **jieba 预热** | 后端启动即加载词典 | 消除首检冷启动 ~1s |

## 三、质量红线(ADR-16:不牺牲准确性)

**Layer2 召回对比(3 查询,思考开 vs 关)**:

| 查询 | 思考开 Top | 思考关 Top | 判定 |
|---|---|---|---|
| 岸桥延迟 | doc_001 | doc_001, +doc_003 | Top 一致,关思考更全 |
| AGV导航 | doc_002 | doc_002 | 完全一致 |
| 5G部署 | doc_003(0.5) | doc_003(0.75), +doc_002 | Top 一致,关思考更全 |

**Layer3 答案对比(岸桥延迟)**:
- 思考开:"视频回传≤50ms,控制信令≤20ms"+ doc_003 相关性说明
- 思考关:"视频回传≤50ms,控制信令≤20ms"(核心事实+引用完全一致,仅少一句次要说明)

**结论**:Top 召回 3/3 一致,关思考反而多找到相关文档(思考开过保守)。质量红线满足。

## 四、可操作性(env 控制)

默认关思考(速度)。操作侧可经 env 开回深度模式:
```bash

SCORE_ENABLE_THINKING=true    # Layer2 开思考(更深判断,~50s)
```

## 六、诚实的边界

- **首检 14-21s 仍非"即时"**——Layer1(3s)+ Layer2 关思考(~10s)+ Layer3(~3s)。要 <10s 需更快模型或更狠的 Layer1 筛选(后续)
- **Layer2 缓存仅 60s**——避免脏数据;长缓存需文档变更失效机制
- **思考关对极复杂推理查询可能略浅**——env 可开回,但默认速度优先(符合"个人办公可用"标尺)
