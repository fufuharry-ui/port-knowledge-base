# Big-Loop #5 — 检索性能优化(实测驱动)

> 4A-BDD-TDD 大循环 #5。**价值标尺(操作侧定)**:个人办公可用——
> 重复查询 P95 < 10s,首检允许偏慢但须有**进度反馈**(不让用户干等)。
>
> **关键:本 loop 基于实测,非假设。** 先测发现瓶颈是"单次 LLM 调用慢"(Layer2 53s + Layer3 25s),
> 而非"调用次数多"(Layer2 已是批量单调用)。优化方向据此修正。

---

## 实测基线(2026-06-30,预热 jieba 后)

| 层 | 耗时 | 说明 |
|---|---|---|
| Layer1 BM25+扩展 | 3.1s | 可接受(本体/实体扩展有开销) |
| **Layer2 LLM 评分** | **53.3s** | 🔴 主瓶颈。单次批量调用(9 候选),但模型慢 |
| **Layer3 LLM 回答** | **24.7s** | 🔴 次瓶颈。非流式,全生成才返回 |
| **总计** | **81s** | LLM 占 78s(96%) |

模型:`qwen3.6-plus`(DashScope 思考模型,慢但准)。无流式输出。

---

## Gate-1:4A 架构

### Business
- **问题**:首次/每次检索 60-80s,是用户流失首要原因(UX 评审定性为落地硬阻塞)
- **价值**:从"能演示"到"能日常用"。P95<10s 是个人办公可用的门槛

### Application(按实测收益排序)
1. **Layer3 真流式输出**(收益最大、风险最低)
   - 现状:`llm_call_text` 非流式 → 等全生成(25s)→ /qa 端点再切片(假流式)
   - 改:Layer3 用 OpenAI stream API 逐 token 返回,首 token < 3s,用户立即看到回答生长
   - **效果**:感知延迟从 25s → ~3s(首 token),总时长不变但体验质变
2. **Layer2 评分提速**(主瓶颈)
   - 方案 A:prompt 严格限定"只输出 doc_id+score,无解释",减少输出 token
   - 方案 B:候选数从 top_k=20 降到 10(Layer1 更狠筛,Layer2 评分对象减半)
   - 方案 C:允许 SEARCH_MODEL 配更快模型(如 qwen-turbo)专用于评分(配置层,不改代码)
3. **jieba 预热**(消除首检冷启动 ~1s,启动时加载)
4. **结果缓存**(短期 memoize,同查询 60s 内复用)

### Data
- 不新增产物,不改 YAML 形状
- 缓存:内存 LRU(进程级),key=查询+index 哈希,TTL 60s

### Technology
- 流式:OpenAI SDK `stream=True` + SSE 透传(/qa 已是 SSE,改 Layer3 返回 generator)
- 缓存:`functools.lru_cache` 或简单 dict + 时间戳
- 测量:计时探针(本 loop 已建),回归守卫

### ADR
- **ADR-15:Layer3 真流式优先于"总时长压缩"**。理由:感知延迟(首 token)对用户体验的决定性
  大于总时长;真流式让 25s 等待变成"立即开始看答案"。风险最低(不改 prompt/模型)。
- **ADR-16:性能优化不改答案质量**。Layer2 候选数压缩需回归测试保证召回不降;
  prompt 精简只去"解释"不去"判断"。
- **ADR-17:缓存只缓存 Layer2+Layer3 结果,不缓存 Layer1**(Layer1 快且依赖实时 index)。
  TTL 短(60s),避免脏数据。

### NFR(可量化,作 Gate-3 ratchet)
- Layer3 首 token 延迟 < 3s(流式)
- 重复查询(缓存命中)P95 < 1s
- 首检总时长不退化(≤ 基线 81s,目标压缩到 < 40s)

## Gate-2:BDD UAT(行为可见)

| ID | 场景 | 可见预期 |
|---|---|---|
| P-1 | 用户提问,Layer3 开始生成 | **首 token < 3s** 出现(答案立即开始生长,非干等 25s) |
| P-2 | 同一查询 60s 内再问 | **< 1s** 返回缓存结果 |
| P-3 | Layer2 评分仍准确 | Top-5 文档与优化前一致(回归守卫,召回不降) |

**UAT 元评审**:P-1(流式感知)是核心价值;P-2(缓存)是高频场景加速;P-3 是质量红线(不能为快牺牲准)。

## 风险与诚实声明
- **Layer2 53s 的根因是 qwen3.6-plus 模型本身慢**——代码层只能压缩输入/输出 token,
  无法改变模型推理速度。若要根本解决,需换更快模型(方案 C,配置层,操作侧定)。
- 本 loop 能做到:**Layer3 感知延迟 25s→3s(流式)+ 缓存命中 <1s + Layer2 适度压缩**。
  首检总时长可能从 81s 降到 ~50-60s,但**达不到 P95<10s**——那需换模型。
- **诚实**:P95<10s 目标在当前模型下**部分可达**(缓存命中场景),首检场景需操作侧决策换模型。
